# Архитектура AI Sales Agent v2

## Цель

Построить AI-менеджера по продажам, который ведет себя как сильный живой sales-менеджер: понимает клиента, знает продукт и документы компании, раскрывает ценность, считает бизнес-кейс, собирает нужные данные и доводит диалог до консультации, оплаты или передачи живому менеджеру.

Система не должна опираться на жесткие сценарии с фиксированными вопросами. Основные решения по ходу диалога должна принимать LLM, а код должен отвечать за данные, инварианты, расчеты, инструменты и аналитику.

## Принцип архитектуры

Нужна не одна LLM с одним большим prompt, а многослойная система:

1. `State Extractor`
2. `Sales Strategist`
3. `Knowledge Retrieval`
4. `Business Calculator`
5. `Response Writer`
6. `Analytics/Event Layer`

Это позволяет разделить понимание, принятие решения, поиск знаний, расчеты и генерацию ответа.

## Основные модули

### 1. Conversation State Layer

Хранит актуальное состояние лида отдельно от сырой истории сообщений.

Что должно храниться:

- имя клиента или preferred name;
- телефон;
- чем занимается бизнес;
- откуда идут заявки;
- объем заявок;
- средний чек;
- скорость ответа;
- потери и конверсия;
- возражения;
- приоритет клиента;
- степень готовности к покупке;
- готовность к консультации;
- желаемая дата и время консультации;
- статус handoff;
- последние рассчитанные бизнес-метрики.

Назначение слоя:

- не заставлять LLM каждый раз заново “догадываться” о состоянии лида;
- отделить структурированные факты от художественной переписки;
- дать базе для аналитики и следующих решений.

### 2. Knowledge Layer

Отдельный слой знаний, из которого LLM получает релевантные фрагменты.

Что входит:

- описание продукта;
- pricing;
- корпоративные документы;
- прайс-листы;
- FAQ;
- кейсы;
- скрипты и регламенты продаж;
- коммерческие условия;
- ограничения и сценарии handoff.

Технически это должен быть retrieval-слой, а не простая подстановка всех файлов в prompt.

Рекомендуемый подход:

- chunking документов;
- embeddings;
- поиск по релевантности;
- возврат только релевантных фрагментов в текущий LLM-вызов;
- метаданные по документам: тип, версия, дата обновления, тема.

### 3. LLM State Extractor

Первый LLM-шаг после каждого входящего сообщения.

Что он делает:

- понимает смысл сообщения клиента;
- извлекает факты в структурированный JSON;
- определяет текущее состояние клиента;
- отмечает новые возражения, интерес, сомнения, сигналы покупки;
- оценивает, достаточно ли данных для следующего шага;
- понимает, были ли названы имя, телефон, время консультации.

На выходе:

- обновление lead state;
- confidence;
- список выявленных фактов;
- список недостающих фактов.

### 4. LLM Sales Strategist

Второй LLM-шаг. Это не генератор текста, а стратег диалога.

Что он решает:

- что сейчас является целью следующего сообщения;
- нужно ли сначала объяснить продукт;
- нужно ли раскрывать ценность;
- нужно ли посчитать бизнес-кейс;
- нужно ли отработать возражение;
- пора ли предлагать консультацию;
- нужно ли сначала собрать имя и телефон;
- нужен ли handoff на человека.

На выходе:

- `next_goal`
- `reasoning_summary`
- `required_tools`
- `should_offer_consultation`
- `should_collect_contact`
- `should_handoff`

### 5. Business Calculator

Это кодовый слой, а не LLM.

Почему:

- деньги и формулы нельзя оставлять полностью на вероятностную модель;
- расчеты должны быть повторяемыми;
- нужна воспроизводимость для аналитики и объяснимость для клиента.

Что должен уметь считать:

- текущие потери бизнеса;
- потенциально возвращаемые заявки;
- ожидаемый рост конверсии;
- влияние на выручку;
- окупаемость внедрения;
- ориентировочную экономию на фонде оплаты труда.

Если данных недостаточно:

- используется набор согласованных допущений;
- все допущения маркируются как ориентировочные.

### 6. LLM Response Writer

Третий LLM-шаг пишет итоговый ответ клиенту.

На вход он получает:

- историю;
- lead state;
- решение стратега;
- найденные знания из документов;
- расчет бизнес-кейса;
- правила бренда и стиля Алины.

На выходе:

- естественный ответ;
- без односложности;
- с живой аргументацией;
- с правильным тоном;
- с учетом этапа продажи;
- с мягким, но уверенным движением к следующему шагу.

## Что должно решаться LLM, а что кодом

### LLM

- понимание клиента;
- извлечение смысла из свободного текста;
- выбор следующего коммуникационного шага;
- работа с возражениями;
- раскрытие ценности;
- построение аргументации;
- формулировка финального ответа.

### Код

- хранение состояния;
- работа с БД;
- stop/do_not_contact;
- handoff;
- запись консультации;
- валидация даты, времени, телефона;
- финансовые расчеты;
- вызов инструментов;
- запись событий в аналитику.

## Booking Flow

Правильная логика записи:

1. Бот раскрывает ценность и доводит клиента до согласия на следующий шаг.
2. Бот спрашивает, когда удобно созвониться.
3. Клиент называет дату и время.
4. Бот проверяет, есть ли имя для обращения.
5. Бот проверяет, есть ли номер телефона.
6. Если данных не хватает, собирает их мягко и по-человечески.
7. Только после этого подтверждает консультацию.
8. После подтверждения отправляет событие админу и в аналитику.

Критично:

- нельзя считать лида записанным, если нет даты и времени;
- нельзя финализировать запись без имени и телефона;
- нельзя переводить в booked только на основе намерения “давайте созвонимся”.

## Event Layer и аналитика

Чтобы потом строить полноценную аналитику, система должна писать не только сообщения, но и события.

Нужные события:

- `fact_extracted`
- `value_explained`
- `objection_detected`
- `objection_handled`
- `calculator_run`
- `consultation_offered`
- `name_collected`
- `phone_collected`
- `booking_time_collected`
- `booking_confirmed`
- `handoff_requested`
- `followup_sent`
- `followup_returned`

Это позволит подключить аналитику не “по догадкам из диалогов”, а по чистым событиям.

## Интеграция с AI Sales Analytics

Проект `AI Sales Analytics` уже содержит полезные блоки, которые можно использовать как основу:

- funnel metrics;
- quality metrics;
- dialog review;
- risky dialogs;
- operational metrics;
- LLM-generated management summary.

Что можно взять как концепцию:

- ежедневный аналитический пайплайн;
- review проблемных диалогов;
- сводки по качеству работы бота;
- выявление узких мест в воронке;
- управленческие рекомендации на основе данных.

Что нужно добавить в бот для сильной интеграции:

- stage events;
- booking events;
- follow-up events;
- handoff events;
- calculator runs;
- snapshots состояния лида;
- журнал решений LLM.

## Целевая формула работы системы

`message -> state extraction -> retrieval -> strategy -> calculation -> response -> event logging`

Именно такая схема должна заменить текущую модель, где одна LLM одновременно и понимает клиента, и принимает решение, и пишет ответ, а код жестко подталкивает ее к следующему вопросу.

## Итог

Целевая версия AI Sales Agent v2 — это LLM-orchestrated sales system, где:

- LLM отвечает за понимание, стратегию и речь;
- код отвечает за данные, расчеты, инструменты и контроль;
- корпоративные документы подключаются через retrieval;
- аналитика строится на событиях и структуре, а не только на тексте переписки.

Такая архитектура даст более живой диалог, более умную логику продаж, более точные расчеты и нормальную управленческую аналитику.
